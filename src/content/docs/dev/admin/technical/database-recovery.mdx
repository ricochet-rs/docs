This page covers how to recover when ricochet fails to start because a database migration trips a foreign key constraint.
The most common cause is having run an unmerged branch build at some point on the same database.

ricochet detects this class of drift automatically.
At startup it compares the live schema against the schema baked into the binary and logs a `WARN` if they disagree.
The `ricochet db doctor` CLI subcommand surfaces the same report on demand — use it as the first diagnostic step before reaching for `sqlite3`.

<Aside type="caution">
  Always take a backup of `RICOCHET_HOME/ricochet.sqlite3` before running any of the SQL on this page.
</Aside>

## Symptom

The pod or service crashes during startup with a message similar to:

```text
Error: while executing migration <timestamp>: error returned from database: (code: 787) FOREIGN KEY constraint failed
```

The migration timestamp varies by release; the constant is the SQLite error code `787` and the phrase `FOREIGN KEY constraint failed`.
Subsequent containers (`fix-permissions`, `merge-config`, etc.) terminate as the main ricochet process exits.

## Why this happens

Some ricochet migrations rebuild a table to widen a `CHECK` constraint or reorder columns.
SQLite implements the rebuild by creating a new table, copying rows in, dropping the old one, and renaming the new one.
At `COMMIT`, SQLite runs a foreign-key check across the entire database.

If any table the released code does not know about holds a foreign key to the rebuilt table, that check fails with error 787 — even when every individual row would still be consistent against the renamed table.

These extra tables almost always come from an unmerged branch build that was deployed against the same database.
The branch's migration created the table; later, a release build's reconciliation removed the migration's bookkeeping row from `_sqlx_migrations`, but the table itself remained.

## Diagnosing

The fastest path is `ricochet db doctor`.
Drop down to the `sqlite3` queries below if the server is unreachable, or you need a wider view than the doctor reports.

### Quick check with `ricochet db doctor`

This step uses the `ricochet` CLI.
Install it first if you don't already have it locally — see [Installing the CLI](/dev/user/quickstart/) — and run `ricochet self update` to make sure you're on a release that includes the `db doctor` subcommand.

```sh
ricochet db doctor -S https://your-ricochet-server
```

The command requires an admin API key (`ricochet login` configures one) and prints a structured drift report.

A healthy database returns:

```text
✓ Database schema matches the expected schema for this server build.
```

A database with branch-build leftovers — the FK 787 scenario — returns something like:

```text
✗ Database schema drift detected.

Phantom tables (in database, not in expected schema)
┌─────────────────────┐
│ Name                │
╞═════════════════════╡
│ deployment_runtimes │
├─────────────────────┤
│ exec_env_checks     │
└─────────────────────┘

Summary: 2 phantom table(s), 0 missing table(s), 0 table(s) with shape drift
```

The `Phantom tables` list is exactly the set you need to drop in the [Recovery](#recovery) procedure below.
Use `--format json` (or `yaml`) to feed the report into other tooling:

```sh
ricochet db doctor -S https://your-ricochet-server --format json
```

The same data is exposed at `GET /api/admin/db/doctor` if you prefer to call the HTTP endpoint directly from an operations script.
The response is the JSON form of the table above and is read-only — it never writes to the database.

### Direct SQL diagnosis

When the server is down (so the doctor endpoint is unreachable), open the database with `sqlite3` and run the following queries.

<Aside type="note">
  These queries are read-only and safe to run against a live database.
</Aside>

**1. Confirm the database is currently consistent.**

```sql
PRAGMA foreign_key_check;
```

Zero rows is expected.
A non-empty result means the database is already inconsistent and recovery requires a different approach (restore from backup).

**2. List every table that holds a foreign key to the table the failing migration is rebuilding.**

The failing migration message names the table.
For example, if the migration rebuilds `deployments`:

```sql
SELECT m.name AS referrer
FROM sqlite_master m, pragma_foreign_key_list(m.name) p
WHERE m.type = 'table' AND p."table" = 'deployments';
```

Compare the result against the tables the released migration explicitly handles.
Any extra entries are phantom referrers from a branch build.

**3. Check `_sqlx_migrations` against the migrations on disk.**

```sql
SELECT version, success, description
FROM _sqlx_migrations
ORDER BY version;
```

Every row should correspond to a file in the release's `migrations/` directory.
A row whose version does not match any file is a stale record from a branch build.

## Recovery

The recovery has three stages: stop the ricochet process, drop the phantom tables as the same UID that owns the database file, and restart ricochet so the pending migration completes.

The example below uses Kubernetes; on bare metal or systemd, the equivalent is to stop the service, run the SQL on the host, then start the service.

<Aside type="caution">
  Dropping a phantom table loses any data in it.
  The released code does not read or write phantom tables, so functional ricochet state is unaffected, but rows from your branch-build experiments will be gone.
</Aside>

<Steps>

1. **Scale the deployment to zero so the SQLite file is not held open.**

    ```sh
    kubectl -n ricochet scale deploy ricochet --replicas=0
    kubectl -n ricochet rollout status deploy/ricochet --timeout=60s
    ```

2. **Override the ricochet container command to `sleep` so the pod can start without running migrations.**

    ```sh
    kubectl -n ricochet patch deploy ricochet --type=json -p='[
      {"op":"replace","path":"/spec/template/spec/containers/0/command","value":["sleep","infinity"]},
      {"op":"replace","path":"/spec/template/spec/containers/0/args","value":[]}
    ]'
    kubectl -n ricochet scale deploy ricochet --replicas=1
    kubectl -n ricochet rollout status deploy/ricochet --timeout=120s
    ```

3. **Exec into the pod and confirm the UID matches the database file owner.**

    ```sh
    POD=$(kubectl -n ricochet get pod -l app.kubernetes.io/name=ricochet -o jsonpath='{.items[0].metadata.name}')
    kubectl -n ricochet exec -it "$POD" -c ricochet-helm -- sh
    id
    ls -ln /data/ricochet.sqlite3
    ```

    The `uid` reported by `id` must match the file owner.
    A mismatch causes SQLite to open the database read-only and `DROP TABLE` to fail with `attempt to write a readonly database`.

    <Aside type="caution">
      If you launch a separate debug pod with a custom image, set `securityContext.runAsUser` to the same UID that owns the database file (commonly `1000` for ricochet helm deployments).
      Otherwise the recovery SQL silently fails to write.
    </Aside>

4. **Back up the database, then drop the phantom tables.**

    The exact list of phantoms comes from the diagnosing step above.

    ```sh
    sqlite3 /data/ricochet.sqlite3 \
      ".backup '/data/ricochet.pre-cleanup-$(date -u +%Y%m%dT%H%M%SZ).sqlite3'"

    sqlite3 /data/ricochet.sqlite3 <<'SQL'
    PRAGMA foreign_keys = ON;

    DROP TABLE IF EXISTS deployment_runtimes;
    DROP TABLE IF EXISTS exec_env_checks;

    PRAGMA foreign_key_check;

    SELECT m.name AS referrer
    FROM sqlite_master m, pragma_foreign_key_list(m.name) p
    WHERE m.type = 'table' AND p."table" = 'deployments';
    SQL
    ```

    `foreign_key_check` should return zero rows.
    The referrer query should return only the tables the released migration handles (for the `deployments` rebuild, that is `logs`).

5. **Restore the original ricochet command and let migrations run.**

    ```sh
    kubectl -n ricochet patch deploy ricochet --type=json -p='[
      {"op":"remove","path":"/spec/template/spec/containers/0/command"},
      {"op":"remove","path":"/spec/template/spec/containers/0/args"}
    ]'
    kubectl -n ricochet rollout status deploy/ricochet --timeout=180s
    kubectl -n ricochet logs -f deploy/ricochet -c ricochet-helm
    ```

    Look for the absence of the FK 787 error and ricochet reaching its normal startup line.

6. **Verify the pending migration completed.**

    Reopen the database and confirm the previously-failing migration is now recorded:

    ```sql
    SELECT version, success, description FROM _sqlx_migrations
    ORDER BY version DESC LIMIT 5;

    PRAGMA foreign_key_check;
    ```

    Both queries should look healthy: the most recent migrations recorded with `success = 1`, no foreign-key violations.

</Steps>

If anything looks wrong at step 6, restore the timestamped backup file before doing further work.

## Prevention

Only upgrade between release tags.
Branch builds may introduce migrations that never make it into a release; their schema effects survive a downgrade or release upgrade and surface as failures during a later table-rebuild migration.

If you need to test a branch build, do so against a copy of the database, not the production file.
