ricochet runs user content — apps and tasks — in an **execution backend**.
Three backends are supported:

| Backend      | `engine` value | Use when                                                                                                          |
| ------------ | -------------- | ----------------------------------------------------------------------------------------------------------------- |
| Host         | (omitted)      | Single host, no container daemon. Content runs as isolated processes on the host via [libcontainer][libcontainer]. |
| Container    | `container`    | Single host with a Docker- or Podman-compatible daemon. Content runs as full containers spawned through the daemon. |
| Kubernetes   | `k8s`          | Multi-node cluster, content scheduled as Pods.                                                                    |

[libcontainer]: https://docs.rs/libcontainer/

If `[backend]` is omitted, ricochet uses the **host** backend.
Content runs as isolated processes managed in-process by `libcontainer` (an OCI-runtime implementation, no daemon required), using the language interpreters installed on the host.
This page documents the `container` and `k8s` backends; the host backend has no configuration knobs of its own.

## Host vs container on a single host

Both backends run content on the same machine as ricochet-server.
The difference is **how** content is isolated and **where** the runtime environment comes from.

| Dimension                | Host backend (`libcontainer`)                                          | Container backend (`container`)                                  |
| ------------------------ | ---------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Isolation mechanism      | `libcontainer` namespaces/cgroups, set up in-process by ricochet-server | OCI containers, spawned by the host's Docker/Podman daemon       |
| Daemon required          | No                                                                     | Yes (Docker, Podman, or compatible)                              |
| Runtime / interpreters   | Whatever is installed on the host (system R, Python, Julia)            | Whatever the chosen image provides; arbitrary per content item   |
| Per-content exec env     | Single shared host environment                                         | Per-item via `exec_env` → image, like the Kubernetes backend     |
| Observability            | Processes are not visible to `docker ps` / container metrics tooling   | Each app/task is a real container — visible to standard tooling  |
| Setup cost               | None — works on a fresh ricochet-server install                        | Install and run a container daemon, grant socket access          |

Pick the **host** backend for simple single-tenant deployments where the host already has the interpreters you need and you do not want to operate a container daemon.

Pick the **container** backend when you need different runtime environments per content item (e.g. R 4.3 vs R 4.5, conflicting system libraries), or when you want each running app and task to be a visible, individually observable container.

## Container backend

Set `engine = "container"` to spawn content as containers via a local Docker- or Podman-compatible daemon.
ricochet talks to the daemon over its Unix socket (`/var/run/docker.sock` by default, or whatever `DOCKER_HOST` points to).

```toml title="ricochet-config.toml"
[backend]
engine = "container"
default_image = "r-default"
exec_env_config_path = "ricochet-exec-env.toml"
content_path = "/var/lib/ricochet/data/content"
cache_path = "/var/lib/ricochet/data/.cache"
```

| Field                  | Required | Description                                                                                                                                                  |
| ---------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `engine`               | yes      | Must be `"container"`.                                                                                                                                       |
| `default_image`        | no       | Image category name from `exec_env_config_path`. Used when a content item does not specify its own exec environment.                                         |
| `exec_env_config_path` | yes      | Path to the exec-env config TOML. Absolute, or relative to `RICOCHET_HOME`.                                                                                  |
| `content_path`         | yes      | Host path bind-mounted into every spawned container at `/opt/ricochet/data/content`. Must contain ricochet's deployment bundles.                             |
| `cache_path`           | yes      | Host path bind-mounted into every spawned container at `/opt/ricochet/cache`. Stores R, Julia, and Python package caches shared across content.              |
| `proxy_host`           | no       | Hostname ricochet's reverse proxy uses to reach spawned content containers. Defaults to `localhost`. See [Reachability](#reachability) below.                |

ricochet refuses to start if `exec_env_config_path`, `content_path`, or `cache_path` is missing — silently launching containers with broken mounts would corrupt deployments.

### Reachability

Each spawned content container publishes its app port to the host's `0.0.0.0`.
ricochet's pingora reverse proxy then connects to that port to route incoming traffic.
The hostname pingora uses is `proxy_host` — `localhost` by default.

That default works in the **systemd-on-host** scenario because ricochet-server and the published port share the host's network namespace.
It does **not** work for **Docker-out-of-Docker**: `localhost` inside the ricochet-server container is that container's loopback, not the host, so the published port is unreachable. See [Deployment scenarios](#deployment-scenarios) below for the fix.

### Image resolution

`default_image` and per-content `exec_env` values are looked up in the exec-env config.
A category like `r-default` resolves to a concrete image reference (`registry.example.com/ricochet/exec-env-r:4.5`).
If the value does not match a category, ricochet treats it as a full image reference and uses it as-is.

### Daemon access

ricochet uses [`bollard`](https://docs.rs/bollard) to talk to the daemon.
Connection target follows standard `DOCKER_HOST` semantics:

- Unset: connects to `/var/run/docker.sock`.
- `unix:///path/to/socket`: connects to that Unix socket.
- `tcp://host:port`: connects over TCP (TLS is not yet configurable — local daemons only).

The user running `ricochet-server` must have read/write access to the socket.
On Linux this typically means adding the user to the `docker` group (or running as root, as the systemd service does by default).

## Deployment scenarios

### ricochet-server via systemd (on the host)

The container daemon and ricochet-server both run on the same host.
ricochet writes deployment bundles to `content_path` on its filesystem, then asks the daemon to bind-mount that same path into spawned containers.
Because both processes see the same filesystem, the host path in `content_path` resolves consistently.

<Tabs>
  <TabItem label="Debian / Ubuntu">
    ```sh
    sudo apt install docker.io
    sudo usermod -aG docker ricochet     # only if ricochet-server runs as a non-root user
    sudo systemctl enable --now docker
    sudo systemctl restart ricochet-server
    ```
  </TabItem>
  <TabItem label="RHEL / AlmaLinux">
    ```sh
    sudo dnf install podman-docker
    sudo systemctl enable --now podman.socket
    sudo systemctl restart ricochet-server
    ```
    Podman exposes a Docker-compatible socket at `/run/podman/podman.sock`.
    Set `DOCKER_HOST=unix:///run/podman/podman.sock` in `ricochet-server.service` if it is not the default socket on your distribution.
  </TabItem>
</Tabs>

### ricochet-server in a container (sibling containers)

ricochet-server itself runs as a container and uses the host's daemon to spawn sibling containers for user content.
This is sometimes called "Docker-out-of-Docker" (DooD).

Three things must line up.

**1. The daemon socket must be reachable from inside the ricochet container.**
Bind-mount `/var/run/docker.sock` into the ricochet container.

**2. Bind-mount paths must resolve to the same physical location on host and inside ricochet.**
The daemon resolves `content_path` and `cache_path` against the **host** filesystem, not the ricochet container's view.
The simplest correct setup is to mount the host data directory into the ricochet container at the same path used in `content_path` / `cache_path`.

**3. `proxy_host` must point to an address ricochet-server can reach.**
Sibling content containers publish their app port to the host's `0.0.0.0`, not to the ricochet-server container's loopback.
Pick one of these (in rough order of preference):

- `--network host` for the ricochet-server container — `localhost` becomes the host's localhost, the default `proxy_host` works without further config. Trades container network isolation for the simplest setup.
- `proxy_host = "host.docker.internal"` — works on Docker Desktop (macOS, Windows) and on Linux Docker when you pass `--add-host=host.docker.internal:host-gateway` to the ricochet-server container.
- `proxy_host = "<host-ip>"` — the host's bridge or LAN IP. Stable but environment-specific.

```sh title="Bridge network with host-gateway alias"
docker run -d \
  --name ricochet-server \
  -p 6188:6188 \
  --add-host=host.docker.internal:host-gateway \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/lib/ricochet:/var/lib/ricochet \
  ghcr.io/ricochet-rs/ricochet-server:latest
```

```toml title="ricochet-config.toml (DooD)"
[backend]
engine = "container"
exec_env_config_path = "ricochet-exec-env.toml"
content_path = "/var/lib/ricochet/data/content"
cache_path = "/var/lib/ricochet/data/.cache"
proxy_host = "host.docker.internal"
```

With matching paths inside and outside, the same `content_path = "/var/lib/ricochet/data/content"` works from either perspective.

<Aside type="caution" title="Path mismatch is silent">
  If `content_path` points to a path that does not exist on the host, the daemon will still launch the sibling container — but bind-mount the wrong (or empty) directory.
  Content items will appear to deploy but fail at runtime because their bundles are not visible.
  Always verify that `ls $content_path` works on the **host**, not just inside the ricochet container.
</Aside>

<Aside type="caution" title="Wrong `proxy_host` is silent too">
  If `proxy_host` resolves but does not actually reach the host (typo, wrong IP, missing `--add-host`), spawned content containers will start and pass health checks at the container layer, but pingora will get a connection refused or timeout when it tries to proxy a request.
  You will see this as 502s in the ricochet UI for the affected content.
</Aside>

See [Container installation](/dev/admin/installation/2-container/) for the full ricochet-server-in-a-container setup.

## Kubernetes backend

For multi-node clusters, use `engine = "k8s"`.
ricochet creates a Deployment per app and a Job per task using the cluster's API.

```toml title="ricochet-config.toml"
[backend]
engine = "k8s"
namespace = "ricochet"
default_image = "r-default"
exec_env_config_path = "/etc/ricochet/exec-env.toml"
pvc_content_name = "ricochet-content"
pvc_cache_name = "ricochet-cache"
image_pull_policy = "IfNotPresent"
```

See [Kubernetes installation](/dev/admin/installation/3-kubernetes/) for cluster prerequisites, Helm chart values, and PVC sizing guidance.

## Remote daemon execution

<Aside type="note" title="Planned">
  Spawning content on a **remote** container daemon — a separate host from where ricochet-server runs — is on the roadmap.
  This requires (a) TLS material configuration for `bollard::Docker::connect_with_ssl`, and (b) a deliberate strategy for `content_path` / `cache_path` since bind mounts resolve on the daemon's host (likely via shared NFS, named volumes, or per-content image baking).
  Until then, use the Kubernetes backend for multi-host execution.
</Aside>
